# 要件定義書

## はじめに

管理者ダッシュボードに日付ごとの利用者人数を集計する機能を追加します。本機能は、既存のTeamViewerアンケートシステムに統合され、アンケート期間中（2026年3月2日～2026年6月27日）の各日付について、その日にTeamViewerを利用したユーザー数を集計・表示します。

## 用語集

- **Daily_Usage_Report**: 日付ごとの利用者人数を集計・表示するレポート機能
- **Report_Generator**: 既存のレポート生成コンポーネント（時間帯別統計を提供）
- **Data_Store**: アンケート回答データを保存するDynamoDB（teamviewer-survey-responsesテーブル）
- **Admin_Dashboard**: 管理者専用のダッシュボードページ（admin.html）
- **Administrator**: レポートを閲覧する管理者（karimata@okijoh.co.jp）
- **Daily_User_Count**: 特定の日付において、いずれかの時間帯でTeamViewerを利用したユニークユーザー数

## 要件

### 要件 1: 日別利用者数の集計

**ユーザーストーリー:** 管理者として、各日付ごとに何人のユーザーがTeamViewerを利用したかを知りたい。日ごとの利用傾向を把握し、利用パターンを分析するため。

#### 受け入れ基準

1. Daily_Usage_Reportは、Data_Storeから全ユーザーの回答データを取得すること
2. 各日付について、Daily_Usage_Reportは、その日のいずれかの時間帯（午前中、午後、18時以降）で利用したユーザーをカウントすること
3. ユーザーが複数の時間帯で利用していた場合、Daily_Usage_Reportは、そのユーザーを1人としてカウントすること
4. Daily_Usage_Reportは、アンケート期間内のすべての日付（2026年3月2日～2026年6月27日）について集計を実行すること
5. 回答データが存在しない日付について、Daily_Usage_Reportは、利用者数を0としてカウントすること

### 要件 2: 日別利用者数のテーブル表示

**ユーザーストーリー:** 管理者として、日別の利用者数をテーブル形式で閲覧したい。日付と人数を一覧で確認し、比較できるようにするため。

#### 受け入れ基準

1. Admin_Dashboardは、日別利用者数を表形式で表示すること
2. テーブルは、日付列、利用者数列、午前中利用者数列、午後利用者数列、18時以降利用者数列の5列で構成されること
3. 日付列は、YYYY-MM-DD形式で表示されること
4. 利用者数列および時間帯別利用者数列は、数値で表示されること
5. テーブルは、日付の昇順（古い日付から新しい日付）でソートされること
6. Admin_Dashboardは、既存のレポート（時間帯別統計、利用パターン分布）の下に日別利用者数テーブルを表示すること

### 要件 2.1: 時間帯別利用者数の集計

**ユーザーストーリー:** 管理者として、各日付ごとに時間帯別の利用者数を知りたい。時間帯ごとの利用傾向を詳細に分析するため。

#### 受け入れ基準

1. Daily_Usage_Reportは、各日付について、午前中（9:00-12:00）に利用したユニークユーザー数をカウントすること
2. Daily_Usage_Reportは、各日付について、午後（13:00-18:00）に利用したユニークユーザー数をカウントすること
3. Daily_Usage_Reportは、各日付について、18時以降に利用したユニークユーザー数をカウントすること
4. 同じユーザーが複数の時間帯で利用していた場合、各時間帯で個別にカウントされること

### 要件 2.2: CSVエクスポート機能

**ユーザーストーリー:** 管理者として、日別利用者数レポートをCSV形式でダウンロードしたい。Excelなどの外部ツールで詳細な分析を行うため。

#### 受け入れ基準

1. Admin_Dashboardは、日別利用者数テーブルの近くに「CSVダウンロード」ボタンを表示すること
2. 管理者が「CSVダウンロード」ボタンをクリックしたとき、システムは日別利用者数データをCSV形式でダウンロードすること
3. CSVファイルは、ヘッダー行（日付、利用者数、午前中、午後、18時以降）とデータ行を含むこと
4. CSVファイル名は、「daily_usage_report_YYYYMMDD_HHMMSS.csv」形式であること（生成日時を含む）
5. CSVファイルの文字エンコーディングは、UTF-8 with BOM（Excelで正しく開けるように）であること
6. dailyUsageデータが存在しない場合、「CSVダウンロード」ボタンは非表示または無効化されること

### 要件 3: 管理者権限の検証

**ユーザーストーリー:** 管理者として、日別利用者数レポートが管理者のみに表示されることを望む。機密情報を保護するため。

#### 受け入れ基準

1. Daily_Usage_Reportは、管理者権限を持つユーザー（karimata@okijoh.co.jp）のみがアクセスできること
2. 管理者以外のユーザーがAdmin_Dashboardにアクセスした場合、Daily_Usage_Reportは表示されないこと
3. 認証されていないユーザーがAdmin_Dashboardにアクセスした場合、システムはログインページにリダイレクトすること

### 要件 4: レポート生成のパフォーマンス

**ユーザーストーリー:** 管理者として、日別利用者数レポートが迅速に表示されることを望む。待ち時間を最小限にするため。

#### 受け入れ基準

1. 管理者がAdmin_Dashboardにアクセスしたとき、Daily_Usage_Reportは10秒以内に生成されること
2. Daily_Usage_Reportは、既存のReport_Generatorと同じリクエスト内で生成されること
3. Daily_Usage_Reportの生成が失敗した場合、システムはエラーメッセージを表示し、既存のレポートは正常に表示されること

### 要件 5: データの整合性

**ユーザーストーリー:** 管理者として、日別利用者数が正確であることを望む。信頼できるデータに基づいて意思決定するため。

#### 受け入れ基準

1. 任意のユーザーの回答データについて、Daily_Usage_Reportは、そのユーザーが利用した日付を正しく識別すること
2. 任意の日付について、Daily_Usage_Reportでカウントされた利用者数は、その日にいずれかの時間帯で利用したユニークユーザー数と一致すること
3. Daily_Usage_Reportの集計結果は、Data_Storeの最新データを反映すること
4. ユーザーが回答を更新した場合、Daily_Usage_Reportは次回のレポート生成時に更新された内容を反映すること

### 要件 6: エラーハンドリング

**ユーザーストーリー:** 管理者として、レポート生成中にエラーが発生した場合でも適切なフィードバックを受け取りたい。問題を認識し、対処できるようにするため。

#### 受け入れ基準

1. Data_Storeへの接続が失敗した場合、Daily_Usage_Reportは「データの取得に失敗しました」というエラーメッセージを表示すること
2. データの集計中にエラーが発生した場合、Daily_Usage_Reportは「集計処理中にエラーが発生しました」というエラーメッセージを表示すること
3. エラーが発生した場合、システムはエラーの詳細をCloudWatch Logsに記録すること
4. エラーが発生した場合でも、Admin_Dashboardの他のレポート（時間帯別統計、利用パターン分布）は正常に表示されること
