# Requirements Document

## Introduction

社内ツールとして、TeamViewerの利用状況を集計するためのアンケートアプリケーションを開発します。本システムはAWS Lambdaの無料枠内で動作し、20〜25名の社員が簡単にアンケートに回答できるWebインターフェースを提供します。

アンケート期間: 2026年3月2日～2026年6月27日

アンケートは1週間単位で表示され、各日付について以下の3つの時間帯の利用状況をクリックで切り替えて入力します：
- 午前中（9:00-12:00）
- 午後（13:00-18:00）
- 18時以降（18:00-）

## Glossary

- **Survey_System**: TeamViewerの利用状況を収集・集計するアンケートシステム全体
- **Web_Interface**: ユーザーがアンケートに回答するためのWebページ
- **Response_Handler**: アンケート回答を受け取り、保存する処理コンポーネント
- **Data_Store**: アンケート回答データを永続化するストレージ
- **Report_Generator**: 集計結果を生成するコンポーネント
- **Authentication_Handler**: ユーザー認証を処理するコンポーネント
- **User**: アンケートに回答する社員（@okijoh.co.jpドメインのメールアドレスを持つ）
- **Administrator**: アンケート結果を閲覧・集計する管理者

## Requirements

### Requirement 1: ユーザー認証機能

**User Story:** ユーザーとして、メールアドレスと共通パスワードでログインしたい。アンケートへのアクセスを社内メンバーに限定するため。

#### 受け入れ基準

1. Web_Interfaceは、メールアドレスとパスワードの入力フィールドを持つログインフォームを表示すること
2. Authentication_Handlerは、入力されたメールアドレスが@okijoh.co.jpドメインであることを検証すること
3. Authentication_Handlerは、入力されたパスワードが事前設定された共通パスワードと一致することを検証すること
4. karimata@okijoh.co.jpでログインした場合、Authentication_Handlerは管理者用パスワード（'Okijoh!admin'）で認証すること
5. 認証が成功したとき、Authentication_Handlerはセッションを作成し、ユーザーをアンケートフォームにリダイレクトすること
6. 管理者として認証が成功したとき、Authentication_Handlerは管理者権限を持つセッションを作成し、管理者ダッシュボードにリダイレクトすること
7. 認証が失敗したとき、Web_Interfaceは「メールアドレスまたはパスワードが正しくありません」というエラーメッセージを表示すること
8. ドメインが@okijoh.co.jpでない場合、Web_Interfaceは「このアンケートは社内メンバー専用です」というエラーメッセージを表示すること

### Requirement 2: アンケート回答機能

**User Story:** ユーザーとして、1週間単位の表形式で利用状況を入力したい。各日付の時間帯をクリックするだけで簡単に回答できるようにするため。

#### 受け入れ基準

1. Web_Interfaceは、認証済みユーザーのみがアクセスできるアンケートフォームを表示すること
2. Web_Interfaceは、1週間分の日付を縦軸、3つの時間帯（午前中、午後、18時以降）を横軸とした表形式で表示すること
3. ユーザーがログインしたとき、Web_Interfaceは過去に入力した回答がある場合、その内容を表に反映して表示すること
4. 各セル（日付×時間帯）をクリックすると、使用/未使用が切り替わること
5. 使用状態のセルは緑色で✓マークを表示し、未使用状態のセルはグレーで-マークを表示すること
6. Web_Interfaceは、週を切り替えるための「前の週」「次の週」ボタンを提供すること
7. アンケート期間外の週には移動できないこと
8. ユーザーがフォームを送信したとき、Response_HandlerはData_Storeに回答とユーザーのメールアドレスを保存すること
9. 既存の回答がある場合、Response_Handlerは新しい回答で既存の回答を更新すること
10. 回答が正常に保存されたとき、Web_Interfaceはユーザーに確認メッセージを表示すること
11. Web_Interfaceは、グリッドのセル幅をヘッダーラベル幅と同じにすること
12. Web_Interfaceは、グリッドに縦罫線と横罫線を表示すること
13. Web_Interfaceは、グリッドのヘッダーラベル（時間帯）を均等配置すること
14. Web_Interfaceは、グリッドの日付をyyyy/mm/dd形式で表示し、曜日も併記すること（例: 2026/03/15 (月)）
15. Web_Interfaceは、ページ全体のフォントサイズを16pxに設定し、読みやすさを確保すること
16. Web_Interfaceは、グリッドのセル高さを38pxに設定し、コンパクトな表示を実現すること
17. Web_Interfaceは、日本時間（JST, UTC+9）を基準として日付を処理すること

### Requirement 2: データ永続化

**User Story:** 管理者として、すべてのアンケート回答が確実に保存されることを望む。2026年3月2日～6月27日の期間にわたる利用パターンを分析できるようにするため。

#### 受け入れ基準

1. Data_Storeは、各アンケート回答をユーザーのメールアドレスをキーとして永続化すること
2. Data_Storeは、各回答の送信タイムスタンプと更新タイムスタンプを記録すること
3. Response_Handlerは、送信から3秒以内に各回答を保存すること
4. Response_Handlerは、同じユーザーからの新しい回答を受け取ったとき、既存の回答を更新すること
5. 保存が失敗した場合、Response_Handlerは最大3回まで操作を再試行すること
6. すべての再試行が失敗した場合、Response_Handlerはエラーをログに記録し、ユーザーにエラーレスポンスを返すこと

### Requirement 3: 回答履歴表示機能

**User Story:** ユーザーとして、自分の過去の回答を確認したい。以前に入力した内容を把握し、必要に応じて変更できるようにするため。

#### 受け入れ基準

1. ユーザーがログインしたとき、Response_HandlerはData_Storeから該当ユーザーの既存回答を取得すること
2. 既存回答がある場合、Web_Interfaceは表のセルに過去の回答を反映した状態で表示すること（使用=緑色✓、未使用=グレー-）
3. 既存回答がない場合、Web_Interfaceはすべてのセルを未使用状態で表示すること
4. Web_Interfaceは、ヘッダー部にログインユーザーのメールアドレスを表示すること
5. ユーザーは、表示された過去の回答をクリックで変更して再送信できること
6. Web_Interfaceは、現在表示中の週のデータをクリアする機能を提供すること

### Requirement 4: 週単位ナビゲーション機能

**User Story:** ユーザーとして、週ごとに回答を入力したい。長期間のデータを見やすく管理できるようにするため。

#### 受け入れ基準

1. Web_Interfaceは、初回表示時にシステム日付を含む週を表示すること
2. システム日付がアンケート期間外の場合、アンケート期間の開始週（2026年3月2日を含む週）を表示すること
3. Web_Interfaceは、「前の週」「次の週」ボタンで週を切り替えられること
4. 現在表示中の週の日付範囲（開始日-終了日）を表示すること
5. アンケート期間の開始日より前の週には移動できないこと（「前の週」ボタンを無効化）
6. アンケート期間の終了日より後の週には移動できないこと（「次の週」ボタンを無効化）
7. 週をまたぐ場合でも、アンケート期間内の日付のみを表示すること

### Requirement 5: AWS Lambda無料枠制約

**User Story:** 管理者として、システムがAWS Lambdaの無料枠の制限内で動作することを望む。運用コストを最小限に抑えるため。

#### 受け入れ基準

1. Survey_Systemは、AWS Lambdaの1GBメモリ制限内で実行すること
2. Survey_Systemは、各リクエストを30秒以内に完了すること
3. Survey_Systemは、無料枠で利用可能なAWSサービスを使用すること
4. Response_Handlerは、コールドスタート時間を5秒未満に最適化すること
5. Survey_Systemは、無料枠の制限を超えることなく、20〜25名の同時アクセスを処理できること

### Requirement 4: 集計レポート生成

**User Story:** 管理者として、集計されたアンケート結果を閲覧したい。組織全体のTeamViewer利用パターンを理解するため。

#### 受け入れ基準

1. Report_Generatorは、管理者権限を持つユーザー（karimata@okijoh.co.jp）のみがアクセスできること
2. Report_Generatorは、受信した回答の総数を計算すること（対象：20〜25名）
3. Report_Generatorは、各時間帯（午前中、午後、18時以降）ごとの利用者数と利用率を集計すること
4. Report_Generatorは、すべての時間帯で利用している人数、一部の時間帯のみ利用している人数を分類すること
5. Report_Generatorは、時間帯別の利用傾向をグラフまたは表形式で表示すること
6. 管理者がレポートを要求したとき、Report_Generatorは10秒以内に結果を生成すること

### Requirement 5: データセキュリティ

**User Story:** 管理者として、アンケート回答が安全に保存・送信されることを望む。従業員情報を保護するため。

#### 受け入れ基準

1. Web_Interfaceは、すべてのデータをHTTPS経由で送信すること
2. Data_Storeは、保存時にデータを暗号化すること
3. Authentication_Handlerは、共通パスワードと管理者パスワードを環境変数として安全に管理すること
4. Survey_Systemは、パスワードをログやレスポンスに含めないこと
5. Response_Handlerは、インジェクション攻撃を防ぐためにすべての入力データをサニタイズすること
6. Survey_Systemは、レポート閲覧をkarimata@okijoh.co.jpアカウントのみに制限するアクセス制御を実装すること

### Requirement 6: アンケート期間管理

**User Story:** 管理者として、アンケートがアクティブな期間を制御したい。2026年3月2日～6月27日の期間のデータを収集するため。

#### 受け入れ基準

1. Survey_Systemは、アンケートの開始日（2026年3月2日）と終了日（2026年6月27日）を設定すること
2. 現在の日付が2026年3月2日より前の場合、Web_Interfaceは「アンケートはまだ開始されていません」というメッセージを表示すること
3. 現在の日付が2026年6月27日より後の場合、Web_Interfaceは「アンケートは終了しました」というメッセージを表示すること
4. 2026年3月2日～6月27日の期間中、Web_Interfaceは新しい回答を受け付けること
5. アンケート期間が終了したとき、Survey_Systemは新しい送信を防ぎつつ、レポート生成は許可すること

### Requirement 7: レスポンシブデザイン

**User Story:** ユーザーとして、任意のデバイスからアンケートにアクセスしたい。デスクトップまたはモバイルデバイスから便利に回答できるようにするため。

#### 受け入れ基準

1. Web_Interfaceは、ビューポート幅1024px以上のデスクトップブラウザで正しくレンダリングされること
2. Web_Interfaceは、ビューポート幅320px以上のモバイルブラウザで正しくレンダリングされること
3. Web_Interfaceは、画面サイズに基づいてフォームレイアウトを適応させること
4. Web_Interfaceは、モバイルデバイス上のすべてのインタラクティブ要素がタッチフレンドリーであることを保証すること（最小44pxのタッチターゲット）
5. Web_Interfaceは、標準的な4Gモバイル接続で3秒以内に読み込まれること

### Requirement 8: エラーハンドリングとロギング

**User Story:** 管理者として、包括的なエラーログを望む。問題をトラブルシューティングし、システムの信頼性を確保するため。

#### 受け入れ基準

1. エラーが発生したとき、Survey_Systemはタイムスタンプ、エラータイプ、コンテキスト情報とともにエラーをログに記録すること
2. Survey_Systemは、監査目的ですべての成功した回答送信をログに記録すること
3. Lambda関数のタイムアウトが発生した場合、Survey_Systemはリクエストの詳細とともにタイムアウトイベントをログに記録すること
4. Survey_Systemは、少なくとも30日間ログを保持すること
5. Survey_Systemは、管理者が日付範囲とエラータイプでログをクエリできる仕組みを提供すること
